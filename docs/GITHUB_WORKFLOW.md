# Downfall GitHub 작업 규칙

## 1. 목적

이 문서는 Downfall 프로젝트에서 디렉터와 AI 개발 어시스턴트가 GitHub 저장소를 안전하고 빠르게 관리하기 위한 기본 규칙을 정의한다.

우선순위는 다음과 같다.

1. 프로젝트 완성도
2. 설정 및 시스템 일관성
3. 장기 유지보수성과 확장성
4. 플레이어 경험
5. 개발 효율성

## 2. 단일 원본과 정보 우선순위

프로젝트 판단은 다음 순서를 따른다.

1. 현재 대화에서 확정된 최신 결정
2. GitHub Wiki 및 프로젝트 문서
3. 디렉터의 추가 지시
4. 일반적인 게임 개발 지식

대화에서 문서보다 새로운 결정을 내린 경우 GitHub 문서는 승인된 작업 범위 안에서 동기화한다. 동기화 전까지는 Pull Request 설명에 불일치와 변경 근거를 기록한다.

## 3. 기본 브랜치 정책

- 기본 브랜치는 `main`이다.
- 일반 변경은 `main`에 직접 커밋하지 않는다.
- AI 작업 브랜치는 다음 규칙을 사용한다.

```text
ai/<영역>/<작업명>-YYYYMMDD
```

예시:

```text
ai/wiki/monster-db-audit-20260705
ai/system/combat-schema-update-20260705
ai/fix/event-id-integrity-20260705
```

## 4. 사용자 요청의 승인 범위

사용자가 다음 표현으로 GitHub 반영을 명시하면 해당 요청은 작업 범위 승인으로 간주한다.

- 수정해줘
- GitHub에 반영해줘
- 브랜치를 만들어서 처리해줘
- 커밋하고 PR까지 만들어줘
- DB를 최신화해줘

범위 승인에 포함되는 작업:

1. 관련 파일 검색 및 읽기
2. 작업 브랜치 생성
3. 요청 범위 내 파일 생성·수정
4. 자동 생성 문서 갱신 준비
5. 검증 도구 실행 또는 CI 구성
6. 커밋
7. Draft Pull Request 생성

별도 승인이 필요한 작업:

- Pull Request 병합
- 파일 영구 삭제
- 대규모 폴더 이동
- Canon 설정의 폐기 또는 근본 변경
- 저장 데이터 호환성을 깨는 스키마 변경
- Force Push 또는 커밋 기록 재작성
- `main` 직접 수정

## 5. 커밋 정책

커밋은 하나의 논리적 변경 단위를 유지한다.

권장 형식:

```text
<type>(<scope>): <summary>
```

주요 type:

- `docs`: 문서
- `feat`: 기능 또는 신규 데이터
- `fix`: 오류 수정
- `refactor`: 구조 개선
- `test`: 검증
- `chore`: 자동화·운영
- `data`: 게임 데이터

예시:

```text
docs(monster): synchronize monster database references
fix(event): align event filename and frontmatter id
chore(ci): validate wiki references on pull requests
```

## 6. Pull Request 정책

AI가 생성하는 PR은 기본적으로 Draft 상태로 만든다.

PR 본문에는 다음 항목을 포함한다.

- 작업 목적
- 변경 파일
- 변경 이유
- 영향 범위
- 검증 결과
- 설정 충돌 또는 미해결 사항
- 디렉터 검토 항목

`main` 병합은 디렉터가 명시적으로 요청한 뒤 수행한다.

## 7. 파일 검색 및 참조 순서

작업 전 다음 순서로 관련 자료를 확인한다.

1. 현재 대화의 최신 결정
2. `Wiki/00_System/Wiki_Index.md`
3. 정확한 문서 ID·파일명·클래스명 검색
4. 단일 출처 문서 원문
5. `depends_on`, `emits`, 데이터 ID의 역참조
6. 관련 코드·이벤트·데이터 교차 검토

DB 또는 ID를 변경할 때는 정의 파일뿐 아니라 모든 참조 파일을 함께 검색한다.

## 8. 단일 출처 원칙

- 상태이상은 상태이상 DB가 소유한다.
- 전투 기술은 기술 카탈로그가 소유한다.
- 괴물 개체는 공유 기술과 상태이상을 ID로 참조한다.
- 캐릭터 고정 설정은 해당 생존자 로어 문서가 소유한다.
- 동일 정의를 여러 문서에 복사하지 않는다.

새로운 데이터 소유권이 불분명하면 구현 전에 소유 문서를 먼저 결정한다.

## 9. 자동 생성 파일

다음 파일은 원본 데이터를 직접 수정해 생성한다.

- `Wiki/00_System/Wiki_Index.md`
- `PROJECT_MANIFEST.md`

자동 생성 파일의 본문을 직접 수정하지 않는다. 생성 스크립트 또는 원본 frontmatter를 수정한 뒤 재생성한다.

## 10. 안전 규칙

- 삭제 전 전체 참조를 검색한다.
- ID 변경 시 마이그레이션 영향과 저장 호환성을 기록한다.
- 여러 핵심 문서를 변경할 경우 하나의 PR에서 영향 범위를 설명한다.
- 검증 실패 상태에서는 병합하지 않는다.
- 불명확한 설정은 확정 설정처럼 기록하지 않고 `신규 제안` 또는 `검토 필요`로 표시한다.

## 11. 작업 종료 보고

모든 GitHub 작업은 다음 형식으로 보고한다.

- 작업 내용
- 변경 사항
- 영향 범위
- 브랜치
- 커밋
- Pull Request
- 검증 결과
- 후속 작업 제안
