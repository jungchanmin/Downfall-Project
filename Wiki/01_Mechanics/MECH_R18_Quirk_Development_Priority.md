# R18 기벽 개발 우선순위 v1.0

## 1. 목적

신규 기벽은 설정 아이디어만으로 추가하지 않는다. 현재 기벽이 어떤 기술 분류와 전술 역할을 강화하는지 먼저 확인하고, 부족한 영역을 보완하는 순서로 개발한다.

```text
신규 기벽 후보
→ 현재 기술 분포 확인
→ 과밀·공백 역할 확인
→ 기존 기벽과 차별성 검사
→ Risk & Return 설계
→ 로그·부위·사건 연결
→ 개발 우선순위 확정
```

## 2. 현재 분포 판단 기준

다음 항목을 함께 본다.

- 기술 분류 연결 수: Service / Tease / Domination / Absorption / Defiance
- 대표 기술 변종 수
- 같은 전술 역할 중복 수
- 연결된 변질 축 수
- 연결된 부위 수
- 일시·지속·영구 기벽 비율
- Risk 강도와 Return 실전성
- 완화 중 정지되는 효과의 범위

단순 연결 수만으로 부족·과밀을 판단하지 않는다. 같은 분류에 여러 기벽이 있어도 역할이 다르면 허용한다.

## 3. 우선순위 등급

### P0 — 시스템 공백 보완

다음 조건 중 하나에 해당한다.

- 특정 기술 분류에 실전 변종이 거의 없음
- 방어·회복·자제 등 필수 역할이 비어 있음
- 특정 로그 축이 기술과 연결되지 않음
- 기존 기벽의 Risk를 활용할 전술 선택지가 없음

### P1 — 과밀 분류의 역할 분산

특정 분류에 기벽이 집중된 경우, 신규 기벽은 다른 분류 또는 다른 역할을 우선한다.

현재 기준에서는 농락형이 가장 과밀하므로 다음 신규 기벽은 원칙적으로 봉사·유혹·흡수·저항형 보완을 우선한다.

### P2 — 기존 역할의 깊이 확장

같은 분류라도 다음처럼 역할이 명확히 다르면 추가 가능하다.

- 안정형
- 제압형
- 역전형
- 회복형
- 자원 교환형
- 자제·탈출형
- 자세 전환형
- 상태이상 대응형

### P3 — 설정·연출 중심 확장

전술적으로 새로운 선택을 만들지 못하고 서사·연출만 늘리는 기벽은 후순위다. 기존 기벽의 이벤트·대사·로그 확장으로 먼저 처리한다.

## 4. 신규 기벽 승인 체크리스트

신규 기벽은 다음을 모두 확인한다.

```yaml
new_quirk_review:
  target_skill_categories: []
  current_category_density:
  intended_role:
  overlapping_quirks: []
  unique_decision:
  risk:
  return:
  memory_axes: []
  memory_parts: []
  mitigation_route:
  proficiency_variant:
  milestone_extension:
```

- 기존 기벽과 다른 의사결정을 제공하는가
- 단순 위력 증가가 아닌가
- Risk와 Return이 같은 상황에서 의미 있게 충돌하는가
- 완화할 이유와 완화하지 않을 이유가 모두 존재하는가
- 관련 로그·부위·사건이 자연스럽게 연결되는가
- Proficiency 대표 변종 1개로 핵심 정체성을 표현할 수 있는가

## 5. 개발 순서 권장

현재 구조 기준 권장 순서:

1. 저항형의 자제·탈출·상태 대응 기벽
2. 봉사형의 관계·지원·비용 교환 기벽
3. 흡수형의 자원 변환·회복 기벽
4. 유혹형의 견제·패링·자세 전환 기벽
5. 농락형의 신규 기벽은 기존 역할과 명확히 다를 때만

이 순서는 절대 고정이 아니다. 기술·기벽 분포가 바뀌면 다시 계산한다.

## 6. 변종 수 제한

- Proficiency: 기벽당 대표 변종 1개
- Milestone: 두 번째 변종 또는 강한 특수 효과 1개
- 같은 기벽이 여러 분류와 연결되더라도 초기에는 하나만 활성 선택
- 추가 변종은 실제 플레이에서 공백이 확인된 뒤 개발
