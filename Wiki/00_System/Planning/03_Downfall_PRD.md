# 📜 Downfall — Product Requirements Document (PRD v1.0)

> **목적**: 위키에 흩어진 내러티브·메커닉 자산을 *코드로 구현 가능한 게임 사양* 으로 압축한 문서. 코딩 단계에 진입하기 전까지 본 문서를 *진리의 원본* 으로 삼는다.

---

## 1. Executive Summary

**다운폴(Downfall)** 은 고립된 마을 *오뱅* 에서 29일을 버티는 1인 플레이용 *턴제 생존 TRPG* 다. 플레이어는 아지트의 리더가 되어 동료들의 광기·자원·시간을 배분하며, 이해 불가능한 외부의 힘과 내부의 인간들 사이에서 균형을 유지해야 한다.

- **장르**: 턴제 생존 / 심리 호러 / 코즈믹 호러 / 자원관리
- **시점**: 2D 정적 / 텍스트 + 일러스트 컴포지션
- **플랫폼 (1차)**: PC (Windows / macOS) — Steam 배포 가정
- **개발 형태**: 1인 개발 + AI 도구 다중 활용
- **예상 플레이타임**: 1회차 6~10시간, 분기 다회차 권장
- **참고 톤**: 셜리 잭슨 「힐 하우스의 유령」 / 「제비뽑기」, 「Darkest Dungeon」 의 시스템 골격

## 2. Vision & Pillars

### 2-1. 게임 비전
> *공포는 외부에서 오지 않는다. 외부의 힘은 단지, 우리가 서로에게 무엇을 할 수 있는지를 드러낼 뿐이다.*

### 2-2. 3대 디자인 필러
| 필러 | 정의 | 게임 내 발현 |
|---|---|---|
| **이해 불가능성** | 플레이어는 끝까지 모든 것을 알 수 없다 | 코즈믹 존재의 정체는 일부만 시사. 이벤트 결과가 이유 없이 빗나간다. |
| **리더의 절대 권한** | 모든 명령은 리더가 내리고, 동료는 따른다 | NPC는 거부 못 함. 단 불만/스트레스/반역으로 *대가* 를 치름. |
| **일상의 배신** | 노골적 공포가 아닌, 미세한 어긋남 | 시각·청각·텍스트가 대부분 평범하다가, 한 디테일만 어긋난다. |

이 세 필러는 향후 모든 구현 결정의 기준이다. **필러를 위반하는 기능은 화려해도 잘라낸다.**

## 3. 타겟 오디언스

| 세그먼트 | 비중 | 후크 |
|---|---|---|
| Darkest Dungeon / Disco Elysium 류 매니아 | 50% | 시스템 깊이, 캐릭터 심리 |

## 4. 코어 게임플레이 루프

```
[Day 시작]
  ↓
새벽:    세계 시뮬레이션 / 외부 위협 이동
  ↓
아침:    자원 배분 / 파견 결정 / 갈등 처리         ← 플레이어 선택
  ↓
점심:    파견조 외부 탐색 / 거점 방어              ← 이벤트 발생 (Track A/B)
  ↓
저녁:    복귀 / 호감도 정산 / 부상·기벽 정산
  ↓
밤:      불침번 / 심리 이벤트 / 휴식               ← 이벤트 발생 (Track A/B)
  ↓
[Day +1, D-카운트 차감]
```

29회의 루프 종료 시 **엔딩 분기**(생존자 명단 / 광기 수치 / 발견한 진실 정도에 따라 6~8개 분기).

## 5. 게임 시스템 명세

### 5-1. 시간/페이즈 시스템
- 5단계 고정 (`Time_System.md` 참조)
- 각 페이즈는 *명시적 종료 트리거* 까지 진행 (모든 행동 선택 완료 시 자동 다음 페이즈로)
- 일부 이벤트는 페이즈를 *건너뛰게* 강제할 수 있음 (특수 트리거)

### 5-2. 자원 시스템
- 핵심 자원: **식량 / 부품 / 금화 / 약품**
- 일일 소모량 = 아지트 인원수 × 종류별 계수
- 결핍 단계: 0%/50%/100%/200% 부족 → 페널티 단계화 (사기 ↓ → 기벽 발현 ↑ → 사망 위험)
- 잉여는 보너스가 아닌 *부패/탈취 위험* 으로 변환 (잭슨식 페널티)

### 5-3. 캐릭터 스탯
| 스탯 | 범위 | 영향 |
|---|---|---|
| 체력 | 0~100 | 0 시 사망 |
| 정신 | 0~100 | 0 시 광기 / 자해 / 도주 |
| 스트레스 | 0~10 | MAX 시 기벽 발현 또는 반역 트리거 |
| 불만도 | 0~10 | MAX 시 반역 트리거 |
| 호감도(리더) | -10~+10 | 반역 동조 판정 가중 |
| 관계도(NPC↔NPC) | -10~+10 | 동조/대립 판정 |

### 5-4. 기벽 시스템
- `Mental_Ailments.md` 의 45종
- 각 기벽: ID / 발현 조건 / 상시 효과 / 이벤트 트리거 키 / 해소 조건
- 한 캐릭터당 동시 보유 권장 ≤ 3개 (4개 시 *붕괴* 상태로 진입)

### 5-5. 이벤트 시스템 (코어 중의 코어)
- **Track A 통보형**: 선택지 없음, 단일 결과. 분위기·세계관 누설용.
- **Track B 개입형**: 2~3 선택지, 각각 성공/실패 분기.
- 이벤트 풀에서 트리거 조건이 만족되는 후보 N개 → 가중치 추첨 → 1개 발현
- 발현 시 게임 상태에 결과 적용 + `#Flag_Memory_*` 발행 (이후 이벤트가 참조)

### 5-6. 메모리/관계 시스템
- 발행된 `#Flag_Memory_*` 는 영구 기록
- 후속 이벤트 트리거 조건 / 분기 / 톤 변형에 사용됨
- UI 상으로 일부만 노출 (플레이어가 *완전한 기억* 을 가질 수는 없음 — 이해 불가능성 필러)

### 5-7. 반역(Mutiny) 시스템
- 트리거: NPC 불만도 또는 스트레스 MAX 도달 시 페이즈 종료에 강제 발생
- 동조자 판정: 다른 NPC들의 (리더 호감도 + 주동자 관계도) 가중합
- 결과:
  - **다수의 반역**: 절반 이상 동조 → 게임오버 (추방 엔딩)
  - **소수의 반란**: 진압 + 리더의 처분 선택 (추방/처벌/도주)

### 5-8. 엔딩 분기
- 생존자 수, 발견된 진실의 비트(코즈믹 존재 정체 노출도), 누적 광기, 반역 횟수 등을 다차원 벡터로 평가
- 사전 정의된 6~8 엔딩 중 가장 가까운 것으로 라우팅

## 7. 데이터 아키텍처

### 7-1. 디렉터리 구조 (런타임)
```
/data
  /events/*.json          ← QA 통과한 이벤트 직렬화본
  /characters/*.json      ← 로어 바이블의 게임 데이터 추출본
  /locations/*.json
  /ailments.json
  /resources.json
  /factions/*.json
  /endings.json
/save
  /slot_N/state.json      ← 플레이어 세이브
```

### 7-2. 핵심 스키마

**Event:**
```json
{
  "id": "EVT_B007_a1f3",
  "track": "B",
  "trigger": {
    "phase": "점심",
    "location": "Loc_상점가",
    "actor_required": ["Gavin"],
    "conditions": [
      { "stat": "스트레스", "op": ">=", "val": 5 },
      { "trait": "통제강박", "actor": "Gavin" }
    ],
    "weight": 1.0
  },
  "narrative": { "title": "...", "stages": [{...}] },
  "choices": [
    {
      "id": "ch1",
      "label": "...",
      "cost": { "time": 1 },
      "check": { "stat": "지능", "dc": 12 },
      "on_success": { "stat_delta": {...}, "flags_emit": [...] },
      "on_fail":    { "stat_delta": {...}, "flags_emit": [...] }
    }
  ]
}
```

**Character (게임 데이터 추출본):**
```json
{
  "id": "Gavin",
  "display_name": "가빈 잭슨",
  "stats_base": { "체력": 70, "정신": 60, ... },
  "traits_initial": ["통제강박"],
  "traits_pool": ["편집증", "강박계산"],
  "relations": { "Rachel": 2 },
  "lore_ref": "/Wiki/03_Entities/Survivors/Guide_Gavin_Jackson.md"
}
```

**Save State:**
```json
{
  "day": 7,
  "phase": "저녁",
  "resources": { "식량": 12, ... },
  "characters": [
    {
      "id": "Gavin",
      "stats_current": {...},
      "traits_active": [...],
      "relations": {...},
      "memory_flags_held": [...]
    }
  ],
  "global_flags": [...],
  "event_history": [{ "id": "...", "day": 3, "outcome": "..." }, ...]
}
```

### 7-3. 트리거 평가 엔진
- 입력: 현재 페이즈, 현재 액터(들), 현재 상태
- 처리: 모든 이벤트의 trigger 조건을 술어 트리(predicate tree)로 평가
- 출력: 후보 풀 → 가중치 정규화 → 추첨

## 8. UI/UX 요구사항

### 8-1. 핵심 스크린
| 스크린 | 기능 |
|---|---|
| 페이즈 헤더 (상시) | Day 진행도 / 현재 페이즈 / D-카운트 |
| 아지트 메인 | 인원 카드 그리드 / 자원 게이지 / 분위기 톤 |
| 캐릭터 상세 | 스탯 / 보유 기벽 / 관계도 / 보유 기억 (일부) |
| 이벤트 모달 | 시네마틱 8단계 텍스트 / 일러스트 / 선택지 |
| 마을 지도 | 15구역 / 위험도 / 파견 슬롯 |
| 일지(Journal) | 발생 이벤트 / 발행 메모리 일부 / 호감도 변화 |

### 8-2. UX 원칙
- **정보 비대칭의 의도적 보존**: 모든 NPC의 모든 스탯을 정확히 보여주지 않는다. 일부는 *추정치* 로만 노출.
- **롤백 금지**: 이벤트 결과 후 되돌리기 없음. 세이브는 *Day 시작 시점* 만 자동 1슬롯.
- **타이포그래피로 분위기 운반**: 본문 폰트는 좁고 회색에 가까움. 크리티컬 모먼트는 약간 어긋난 자간.
- **음향 페이즈 큐**: 페이즈 전환 시 짧은 앰비언트 스팅거.

## 9. 기술 사양

### 9-1. 권장 엔진
**Godot 4 (GDScript + C# 옵션)** — 1인 개발 + AI 코딩 도구 친화도가 가장 높음.
- 대안: Unity (생태계 풍부하지만 빌드 복잡도 ↑)
- 대안: Pixi.js + Electron (웹 친화적, 텍스트 중심에 적합)

→ **1차 선택: Godot 4**.

### 9-2. 레이어 구조
```
[Presentation Layer]   Godot Scene/Node
       ↓
[Application Layer]    GameController / EventDirector / TurnManager
       ↓
[Domain Layer]         Character / Resource / Event / Memory
       ↓
[Data Layer]           JSON 로더 / 세이브 시스템
```

### 9-3. 외부 의존성 최소화
- JSON 파싱: 엔진 내장
- 직렬화: 내장 + 자체 마이그레이션 함수
- 음향: 엔진 내장
- 폰트: 라이선스 자유 폰트만

### 9-4. 빌드 타깃
- 1차: Windows x64
- 2차: macOS (ARM/x64)
- 3차: Linux

## 11. 리스크 & 대응

| 리스크 | 발생 영향 | 대응 |
|---|---|---|
| **이벤트 톤 표류** | 분위기 붕괴 → 핵심 매력 상실 | QA 프로토콜 Gate B 강제, 디렉터 1인 검수 |
| **시스템 과적재** | 1인 일정 파탄 | 필러 위반 기능 칼질 정책 |

| **밸런스 광역 망가짐** | 후반 출시 임박 시 손볼 곳이 폭주 | M2 시점에 자동 시뮬레이터 구축 |
| **장르 좁음에 따른 시장 리스크** | 매출 부진 | 데모 + 시각적 커뮤니케이션 강화, 인플루언서 타겟팅 |

## 12. 성공 지표 (Launch + 3 months)

- **Steam 평가** Mostly Positive 이상 (≥80%)
- **평균 플레이 타임** ≥ 6시간 (게임당 1회차 분량)
- **재플레이율** ≥ 25% (다회차 분기 효과 검증)
- **데모 → 정식 전환율** ≥ 30%

---

## 13. Out of Scope (1차 출시에 포함하지 않음)

- 멀티플레이
- 모바일 포팅
- 모드 도구
- 절차적 이벤트 생성 (인게임)
- 음성 더빙
- DLC

이들은 M7 이후의 *포스트런치* 안건으로 별도 검토.
