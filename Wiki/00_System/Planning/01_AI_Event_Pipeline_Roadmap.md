# 🏭 AI 이벤트 생성 파이프라인 로드맵 (v1.0)

> **목적**: `EVT_Conveyor_Belt_Master_Prompt.md` 가 안정적으로 작동하기 위해 위키의 어떤 항목을 어떤 순서로 채워야 하는지를 정의한 마스터 빌드 오더(Build Order).
>
> **원칙**: 컨베이어 벨트는 자기 자신을 부팅할 수 없다. 컨베이어 벨트가 호출하는 모든 *데이터 종속성(Data Dependency)* 이 먼저 존재해야 한다.

---

## 1. 종속성 분석 (Dependency Map)

`EVT_Conveyor_Belt_Master_Prompt.md` 의 5단계는 다음 데이터를 연쇄적으로 호출한다.

```
Step 1 (요청)        →  Wiki_Index.md (있음)
Step 2 (브레인스토밍)  →  EVT_Notification.md / EVT_Interactive.md (양식)
                     →  Bot_Universal_Template.md (캐릭터 해부 양식)
Step 3 (뼈대 설계)    →  Mental_Ailments / Resource_Logic / Time_System (수치/규칙)
                     →  Oebeng_Village_Lore (장소)
                     →  Survivors/*.md (각 캐릭터 로어 바이블)
Step 4 (지침 점검)    →  ConText_Manifest.md (있음, 라이팅 지침)
Step 5 (출력)        →  05_Events/ (출고 위치)
```

→ 빌드 오더는 **하위(데이터) → 상위(이벤트)** 순서를 엄격히 따라야 한다. 캐릭터 로어 바이블이 비어 있는데 캐릭터 이벤트를 양산하면, 페르소나가 표류하여 톤이 무너진다.

## 2. Phase별 빌드 오더

### 🟢 Phase 0 — 시스템 부팅 (DONE)
| 산출물 | 상태 | 비고 |
|---|---|---|
| `00_System/Wiki_Index.md` | ✅ | 인덱스 |
| `ConText_Manifest.md` | ✅ | 라이팅 지침/태그 스키마 |
| `00_Templates/EVT_Conveyor_Belt_Master_Prompt.md` | ✅ | 마스터 프롬프트 |
| `00_Templates/Bot_Universal_Template.md` | ✅ | 캐릭터 해부 양식 |

### 🟡 Phase 1 — 템플릿 정합성 확보 (1순위, 즉시 실행)
이벤트 양식은 **출력 스키마**다. 이게 흔들리면 모든 후속물이 흔들린다.

| 작성/검증할 산출물 | 핵심 검증 항목 |
|---|---|
| `00_Templates/EVT_Notification.md` | Track A 헤더 / 트리거 조건 / 본문 / 즉시 효과(Stat·Trait·Flag) 필드. 선택지 없음 명시. |
| `00_Templates/EVT_Interactive.md` | Track B 헤더 / 트리거 / 본문 / **2~3 선택지** / 분기별 성공·실패 판정 / 결과 효과 / 흔적·기억 플래그 필드. |
| 두 템플릿 공통 | 시네마틱 8단계 슬롯, 셜리 잭슨 4지침 체크박스, `#Tag` 명명 규칙 준수 예시. |

**Done의 정의**: 둘 중 하나의 템플릿에 임의의 캐릭터·장소를 끼워 넣어도 빈칸 없이 채울 수 있어야 한다.

### 🟡 Phase 2 — 메커닉 데이터 확정 (2순위)
이벤트의 **수치 결과**가 의지하는 진리값.

| 산출물 | 채워야 할 핵심 |
|---|---|
| `01_Mechanics/Mental_Ailments.md` | 001~045 기벽. 각 항목: ID·이름·발현 조건·상시 페널티·이벤트 트리거 키·해소 조건. |
| `01_Mechanics/Resource_Logic.md` | 식량/부품/금화 일일 소모량, 결핍 시 페널티 단계, 잉여 시 보너스. **이벤트가 ±몇 단위까지 줄 수 있는지 상한선** 명시. |
| `01_Mechanics/Time_System.md` | 5페이즈 각각의 행동 종류·소요·금기 행동. 페이즈 종료 트리거 조건. |

**Done의 정의**: 이벤트 작성자(AI)가 "스트레스 +3"을 적었을 때, 그게 임계점 대비 얼마나 큰 일인지 판단 가능해야 한다.

### 🟡 Phase 3 — 월드/장소 데이터 (3순위)
이벤트 트리거의 **공간 좌표**.

| 산출물 | 채워야 할 핵심 |
|---|---|
| `02_World/World_Concept.md` | 29일 카운트다운 구조, 코즈믹 호러 신화 맥락, 3대 지주 세력 정의. |
| `02_World/Oebeng_Village_Lore.md` | 15개 구역 각각의: 위험도·자원 풍부도·고유 사건 시드·분위기 키워드·셜리 잭슨식 "공간의 악의" 한 줄. |

**Done의 정의**: 어떤 구역에 누구를 보내든, 해당 구역의 이벤트 시드가 최소 3개 이상 떠올라야 한다.

### 🟠 Phase 4 — 캐릭터 로어 바이블 (4순위, 가장 큰 노동)
**가장 시간이 많이 드는 단계.** 한 캐릭터의 로어 바이블 부재는 그 캐릭터의 모든 이벤트 품질을 무너뜨린다.

작성 우선순위:
1. **리더 후보군 (Player-facing)** — 가빈, 레이첼 ✅ 진행 중
2. **초기 아지트 동료 NPC** — 4~6명 (29일 시작 시점에 이미 존재하는 인물)
3. **합류 가능 NPC** — 마을에서 영입 가능한 인물 (3~5명)
4. **갈등 축 NPC** — 반역/배신/광신도 동조 가능성을 가진 인물

각 인물당 `Bot_Universal_Template.md` 풀 필드 작성. 특히:
- `Declared` vs `Actual` 간극 (모순의 연출 재료)
- `Primary Setting` (공간의 유기적 결합 재료)
- 보유 기벽 ID 1~3개 (`Mental_Ailments` 와의 연결)

**Done의 정의**: 임의의 두 캐릭터를 같은 장소에 던져도, 둘이 어떤 식으로 부딪힐지 즉답 가능해야 한다.

### 🟠 Phase 5 — 팩션/적대자 데이터 (5순위)
| 산출물 | 채워야 할 핵심 |
|---|---|
| `03_Entities/Factions/` | 3대 지주 세력: 각각의 교리·자원·접촉 시 이벤트 시드. |
| `03_Entities/Antagonists/` | 코즈믹 존재(들), 광신도, 마을의 변질된 잔존자. **노골적 묘사 금지, "일상의 배신" 형식으로 묘사**. |

### 🔴 Phase 6 — 파일럿 이벤트 양산 (6순위)
이제서야 컨베이어 벨트를 실가동.

| 단계 | 작업 |
|---|---|
| 6-1 | **파일럿 5종**: Track A 2개, Track B 3개. 가빈/레이첼/3개 장소 조합. |
| 6-2 | `Event_QA_Protocol.md` (별도 문서)로 풀 검수. 합격선 통과 시에만 다음으로. |
| 6-3 | **장르 샘플러 20종**: 페이즈별·캐릭터별·장소별 분포 점검. |
| 6-4 | 분포 갭 분석 — 어떤 캐릭터·장소·기벽이 이벤트 수가 부족한지 식별. |

### 🔴 Phase 7 — 본격 양산 (7순위)
| 차원 | 양산 목표 |
|---|---|
| 일자별 (Day 1~29) | Day마다 "고정 이벤트" 2~4개 + "랜덤 풀" 5~10개 |
| 페이즈별 | 5페이즈 × 평균 2~3개 트리거 가능 풀 |
| 캐릭터별 | 캐릭터 1인당 고유 이벤트 최소 8개 (탐색 3 / 갈등 2 / 기벽 발현 2 / 호감도 1) |
| 장소별 | 장소 1곳당 고유 이벤트 최소 4개 |
| 특수 | 반역 트리거 / 광신도 합류 분기 / 엔딩 분기 이벤트 |

### 🟣 Phase 8 — 통합 패스 (마지막)
- 이벤트 간 **참조 무결성 검증** (`#Flag_Memory_X` 가 발행만 되고 어디서도 소비되지 않는 경우 색출).
- 이벤트 그래프 시각화 (어느 이벤트가 어느 이벤트를 잠금 해제하는지).

## 4. 위험 요소 (Build Risks)

| 위험 | 증상 | 대응 |
|---|---|---|
| **로어 바이블 표류** | 캐릭터마다 톤이 다른 작가가 쓴 듯함 | Phase 4를 한 사람(또는 한 세션)이 몰아서 작업 |
| **태그 인플레이션** | 새 이벤트마다 새 `#Stat_` 추가 | `ConText_Manifest.md` 의 태그 명명 규칙을 락(lock)으로 취급. 신규 태그 도입 시 별도 결재. |
| **공간 균질화** | 모든 장소가 똑같이 무서움 | Phase 3 작성 시 "이 장소만의 한 줄"을 강제 |
| **선택지 더미화** | Track B에서 한쪽 선택지가 명백히 우월해 의미가 없어짐 | Phase 1 템플릿에 "각 선택지의 기회비용" 필드 강제 |
