# Downfall 문서 소유권 및 폴더 규칙

## 1. 기본 원칙

- Wiki에는 게임이 직접 참조하는 세계관, 시스템, DB, 이벤트와 시나리오 정본을 둔다.
- `docs/`에는 개발 절차, 로드맵, 프로젝트 상태, 도구 사용법과 저장소 운영 문서를 둔다.
- 같은 규칙을 여러 문서에 복제하지 않는다.
- 상위 문서는 개요와 불변 규칙, 하위 문서 탐색 경로를 소유한다.
- 상세 계산, 예외, 알고리즘과 개별 데이터는 세부 문서 또는 DB 한 곳에서만 소유한다.

## 2. 목표 폴더 구조

```text
Wiki/
├─ 00_System/
├─ 01_Mechanics/
├─ 02_World/
├─ 03_Entities/
├─ 04_Items/
├─ 05_Events/
├─ 06_Narrative/
├─ 90_Modules/R18/
└─ 99_Archive/

docs/
├─ governance/
├─ planning/
├─ development/
└─ ai/
```

본 구조는 목표안이며 기존 파일은 검토 없이 일괄 이동하지 않는다.

## 3. 문서 유형별 소유권

### 프로젝트 정본

프로젝트 정의, 디자인 필러, 절대 세계 규칙과 문서 우선순위를 소유한다.

### 상위 시스템 문서

시스템 목적, 핵심 흐름, 불변 규칙, 통합 계약과 세부 문서 참조표를 소유한다.

예: `MECH_Combat_System.md`는 원형을 유지하되 상세 계산과 DB를 직접 중복 기재하지 않는 전투 허브로 발전시킨다.

### 세부 시스템 문서

복잡한 내부 규칙, 계산, 상태 전이, 예외, 알고리즘, 구현 주의사항과 테스트 사례를 소유한다.

### DB

개별 기술, 상태이상, 괴물, 아이템, 기벽과 밸런스 값을 소유한다.

### 이벤트 및 시나리오

장면, 조건, 선택, 판정, 결과, 기억과 후속 연결을 소유한다. 시스템 정의를 이벤트 파일 안에서 새로 만들지 않는다.

## 4. 일반 시스템과 R18 모듈

의존 방향은 다음을 기본으로 한다.

```text
Core 시스템 ← R18 모듈
Core 시스템 ✕ R18 전용 문서 직접 의존
```

Core에 남기는 요소:

- 친밀도
- 관계 상태
- 기억
- 침식도
- 욕구불만
- 동의·거부 및 선택지 조건
- NPC 성향과 상태

R18 모듈로 분리할 요소:

- 성인 전용 이벤트와 시나리오
- 성인 기술과 상태이상
- 성인 괴물 행동
- 성인 패배 결과
- 성인 묘사 규칙과 갤러리

완전 분리가 캐릭터 정합성을 훼손할 경우 Core 문서에는 공통 성향만 남기고 R18 확장 문서가 추가 조건과 장면을 소유한다.

## 5. 세력 및 적대 집단 분류

- `Faction`: 지속 관계, 호감도, 거점과 거래를 가진 정식 세력
- `HostileGroup`: 전투, 약탈과 납치를 중심으로 하는 비정규 집단
- `NamedAntagonist`: 고유 서사와 반복 출현을 가진 개별 적대자
- `InternalThreat`: 아지트 내부의 배신, 범죄와 갈등

## 6. 이전 절차

1. 신규 정본 문서 작성
2. 기존 문서와 충돌 목록 작성
3. 목적지와 문서 소유권 승인
4. 파일 이동 및 참조 갱신
5. Wiki Index와 Manifest 재생성
6. CI 검사
7. 기존 문서를 deprecated 또는 Archive 처리

삭제, 대규모 이동, ID 변경과 기존 문서 덮어쓰기는 별도 승인 후 수행한다.

## 7. 경로와 ID

- 경로는 분류와 탐색을 위한 구조다.
- 문서 간 의존성은 파일 경로가 아니라 고유 `id`를 사용한다.
- 파일 이동 시 가능한 한 기존 ID를 유지한다.
- 문서 역할이 완전히 달라지는 경우에만 새 ID를 발급한다.

## 8. 레거시 문서

frontmatter가 없거나 최신 정본과 충돌하는 문서는 자동으로 정본화하지 않는다. `docs/WIKI_MIGRATION_BACKLOG.md`에 기록하고 개별 검토 후 등록, 이동, 통합 또는 폐기한다.
